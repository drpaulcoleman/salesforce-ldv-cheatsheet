01 — What Is LDV? — The 4-Lever Mental Model
📊 LDV Definition (Salesforce Official Triggers)
An org has Large Data Volume characteristics if any of the following apply:
  • Tens of thousands of users
  • Tens of millions of records on a single object
  • Hundreds of GB of total record storage
The numbers are not magic. An org of 5M Cases and ten heavy integration users can show LDV symptoms long before reaching official triggers. Symptom-driven assessment beats gates.
Symptoms That Force the Discussion
  • Reports/list views time out (QUERY_TIMEOUT, 120s default)
  • Bulk DML produces UNABLE_TO_LOCK_ROW
  • Sandbox refresh takes > 24h
  • Single-record save > 5s (Lightning page TTI penalty)
  • Async jobs queue up (CONCURRENT_REQUESTS_LIMIT_EXCEEDED)
🧠 The 4-Lever Mental Model
Every LDV problem is one (or more) of these four levers being out of place. Pull them in order.
#LeverQuestionWhere
1SelectivityDoes every query hit an index returning < 10% of rows?§08–§12
2SkewAre there parents/owners holding > 10K children/records?§22
3AsyncIs heavy work outside user-blocking transactions?§23, §28
4ArchiveIs the working data set as small as it can possibly be?§27
Working order: Fixing skew on a non-selective query just moves the bottleneck. Always fix selectivity first. Async-ing a query that does a full table scan just hides the cost — the org still pays.
🔥 CORE PHILOSOPHY — Salesforce Is a CRM, Not a Big Data Platform
🚀 Keep only the most ACTIONABLE data hot. Archive or virtualize the rest. The Salesforce Lightning Platform is a transactional system of engagement and a system of record for actively-managed customer relationships. It is NOT a data lake, NOT a warehouse, NOT a low-cost archive, and NOT a reporting backend for 10-year-old data. Every record you keep "just in case" pays a daily tax in slower list views, longer sharing recalcs, fatter sandboxes, more lock contention, and dimmer Optimizer scores.
Hot (in Salesforce)
  • Active opportunities, open cases, current quarter
  • Customer-facing data (LWC pages, mobile)
  • Records actively edited or routed by automation
  • Last 12–24 months of activity
Warm (Big Objects / Connect)
  • Audit trails, compliance history
  • Closed cases > 1–2 years old
  • Historical orders / line items
  • Denormalized aggregates for analytics
Cold (External / Heroku Connect / Lake)
  • Years-old transactional ledgers
  • High-volume IoT / clickstream
  • Marketing event firehose
  • Anything queried < 1×/quarter
Litmus test: If a record was last edited > 18 months ago, last viewed > 6 months ago, and isn't on a regulatory hold — it doesn't belong in your hot org. Archive to a Big Object or external store, surface via Salesforce Connect / External Objects when needed.
Companion architectural constant — the Rule of 200: the platform internally chunks almost every record-bearing operation (triggers, Bulk API, Batch Apex, SOAP create/update, the implicit Apex SOQL for-loop) into 200-record batches. It's the implicit unit of "bulk" on Salesforce. Combined with the hot/cold philosophy above, it forms the platform's two foundational architectural constraints: keep the dataset lean, and design every record-processing surface to behave correctly with 200-at-a-time chunks. Full treatment in §16.

🔍 Why Custom Fields Need Index Tables
Custom field data lives in a generic VALUE column — type-erased and unindexable in place. Salesforce builds a separate index table per indexed custom field.
Native heap rowValue0='A', Value1='B', Value2=NULL ...
Index table row(typed) Status__c='Closed' → recordId
Two Critical Side-Effects
  • Index tables exclude NULL rows by default — WHERE field = null can't use the index unless Support enables Null-Inclusive indexing.
  • A formula field that resolves to NULL behaves the same way as a real null. Pre-populate with a sentinel value if you need the rows indexed.

03 — The Lightning Platform Query Optimizer — The Engine Behind Every SOQL
💡 Orientation — What This Section Is (and Isn't):
This region: the ENGINE
The Lightning Platform Query Optimizer is a platform-internal component that automatically runs every time any SOQL, report, list view, or API query is executed. It is not a tool you invoke. You cannot turn it off, run it on demand, or hand-tune its decisions. This region explains how it works — what decisions it makes, what data it consults, and what governs its behavior — so that an architect can reason about why a query is fast or slow before reaching for the diagnostic tool.
The window: §30 — SOQL Explain
When you run Query Performance Feedback (SOQL Explain) in §30, you're asking the optimizer to tell you the plan it would have chosen for a specific query. §30 is the diagnostic window into this engine's decisions. The two sections are complementary: §03 explains why the engine behaves the way it does; §30 shows you what it decided for your specific SOQL. Without §03, §30's output is just numbers.
All the selectivity rules in §09, anti-patterns in §10, and AND/OR/LIKE behavior in §11 are properties of this engine. Understanding it once explains every threshold and every rewrite pattern in the rest of the document.
⚙ What the Engine Decides — Per Query, Automatically
For every SOQL, report, list view, or API query() the platform receives, the optimizer makes these decisions before any rows are fetched:
  • Picks the leading index from the WHERE filters, if one of them qualifies for index use under the selectivity rules (§09).
  • Picks the leading driving table when no filter qualifies for an index path. Worst case: full scan.
  • Orders the remaining tables / joins to minimize total cost (object joins, sharing joins, foreign-key traversals).
  • Injects platform-managed sharing joins automatically for record-visibility enforcement, unless WITH SYSTEM_MODE is specified (§17).
  • Consults and updates internal statistics tables to make all of the above decisions.
The Pre-Query — How the Engine Avoids Using Bad Indexes
Before running the actual SOQL, the optimizer queries an internal statistics table to count the expected match count for each indexed filter. If the count exceeds the selectivity threshold, the index is skipped — even though the field is indexed — and the optimizer falls back to a full driving-table scan. This is why "indexed" is not the same as "uses the index." See §09 for the threshold ladders that govern this decision.
🕒 Why The Engine Sometimes Picks "Wrong" After A Big Load
The optimizer's decisions are only as good as the statistics it consults. Salesforce maintains its own statistics (the underlying RDBMS's native stats are insufficient for the multi-tenant model — tenant-scoped row counts must be tracked separately).
Refresh cadence: nightly. After a massive data load (> 1M rows), the optimizer is working from stale statistics until the next nightly stats run completes — and may pick suboptimal plans during that window.
Practical Consequences
  • Run smoke tests / production-shape Query Plan checks (§30) the day after a major load, not the same day.
  • For data migrations, schedule cutover after the nightly stats refresh has run on the loaded data.
  • When loading 10M+ records via Bulk API, expect a 24–48h "wobble" before query plans stabilize.
  • If a previously-fast query suddenly slows down right after a load, suspect stale stats first — before refactoring.
📊 Inspecting Selectivity Yourself — Before The Engine Has To
The optimizer's pre-query consults internal stats you can't see directly — but you can approximate the same decision ahead of time using GROUP BY ROLLUP from anonymous Apex or Workbench. Use this before writing the production filter, and as a baseline before requesting a custom index from Support.
// Per-value distribution — tells you which values are unselective
SELECT Stagename, COUNT(Id)
FROM Opportunity
GROUP BY ROLLUP(Stagename)

// Combined-criteria distribution — the AND intersection
SELECT WEEK_IN_YEAR(CloseDate), CALENDAR_YEAR(CloseDate), COUNT(Id)
FROM Opportunity
WHERE IsDeleted = false
GROUP BY ROLLUP(WEEK_IN_YEAR(CloseDate), CALENDAR_YEAR(CloseDate))
Why IsDeleted = false: ROLLUP queries include soft-deleted Recycle Bin records by default. Filter them out for an accurate active-data picture — the optimizer's pre-query also excludes them when sharing/visibility allows.
Workflow: use ROLLUP to discover unselective values → rewrite filters to use selective ones (or stamp a real selective field per §10's formula-field treatment) → verify the optimizer's chosen plan via R27's Query Explain.

⚠ The Meta-Anti-Pattern: Coding Around Governor Limits. Custom Apex / triggers / SOQL gymnastics designed to defeat a governor limit (chunked recursion, sleeping, retrying, splitting batches by hand) is a smell that the design is wrong. Governor limits are a signal that the work belongs somewhere else — async, archived, or off-platform. Always prefer OOTB / low-code (Flow, Reporting Snapshot, Big Object) before custom Apex; prefer Apex before "clever" Apex; never write Apex whose only purpose is to dodge a governor.
05 — Reports, List Views, Triggers & Flows at LDV — Anti-Patterns
Reports / List Views
  • "All Time" reports on LDV objects — force rolling windows
  • Cross-object reports across 4+ joined objects
  • Filters on non-indexed custom fields
  • Filters using formula fields (cross-object refs especially)
  • Sort by long-text or unindexed field
Mitigations
  • Filters on CreatedDate, SystemModstamp, OwnerId, RecordType
  • Default list view to "My active records, this quarter"
  • For aggregations across LDV: use CRM Analytics (Tableau CRM) or pre-computed summary objects
  • Limit dashboard components per page
  • Move heavy historical reports to a Big Object + paginated UI
Triggers, Flows, Process Builder
  • SOQL or DML inside a loop
  • Multiple triggers on the same object (no order guarantee)
  • Process Builder on LDV objects — deprecated & slow; migrate to Flow / Apex
  • Roll-up summary chains across LDV (cascading parent locks)
  • Trigger → Flow → Trigger cascade with no guard
  • Recursion not guarded by static processed-Id Set
Patterns
  • One trigger per object + handler-class framework
  • Bypass switch via Custom Metadata / Settings (for loads)
  • SOQL filtered by Trigger.newMap.keySet()
  • Queueable for follow-up DML on parents (avoid lock storms)
  • DLRS or scheduled Apex for roll-ups > LDV threshold

🔒 Open Database Cursors — the Integration-User Gotcha
A database cursor is a server-side handle the platform allocates to track an in-progress query result set so the client can fetch additional pages. SOAP queryLocator (paged via queryMore()), REST query pagination (paged via nextRecordsUrl), Bulk API 2.0 query result sets, and Apex Cursors (Database.getCursor(), Spring '24+) each hold an open server-side cursor until it's consumed to completion or times out. None of these surfaces expose an explicit client-side close API — lifecycle is managed by exhaustion or expiration. Salesforce caps how many can be open per user, per org.
Approximate caps (verify against current docs):
  • Per user, per org: ~10 open database cursors across SOAP queryLocator + REST query pagination + Bulk API result sets
  • Apex Cursors (Database.getCursor(), Spring '24+) and Pagination Cursors share these per-user expiration limits with API Query cursors; per-transaction and per-day instance budgets are separate (see Limits.getApexCursors(), Limits.getApexPaginationCursors())
  • Default cursor lifetime before automatic cleanup: ~15 minutes idle
The architecture trap: When five services all integrate via one shared "Integration User" sharing a session token, they share the same per-user database-cursor pool. Service #6 to start a Bulk query gets "Too many query cursors open". Symptoms appear "randomly" in production because the failure depends on whether other services happen to be mid-paginate at the same time.
Mitigation
  • One integration user per integrating system — preserves identity attribution AND each system gets its own database-cursor budget
  • For high-fan-out integrations: a pool of integration users, modulo on system-id, so cursor pressure is sharded
  • Consume to completion — iterate queryMore() / follow nextRecordsUrl until done; don't abandon partial result sets and wait on the 15-minute idle timeout to free the cursor
  • Monitor via Event Log (BulkApiResultEvent, ApiEvent) for cursor lifecycle — correlate slow queries with cursor allocation
  • For Apex Cursors: cursors are stateless and platform-managed — there is no close() method. Track Cursor.fetch(position, count) offsets yourself; the cursor is released by exhaustion or expiration

08 — Indexed Fields, Custom Indexes & What Cannot Be Indexed
✅ Auto-Indexed Out of the Box
All Objects (Standard Indexes)
  • Id (primary key)
  • Name
  • OwnerId
  • CreatedDate
  • SystemModstamp / LastModifiedDate
  • RecordType / RecordTypeId
  • Division
  • All master-detail and lookup foreign keys
  • Email on Contact and Lead
Auto-Indexed Custom (No Support Needed)
  • Fields marked Unique
  • Fields marked External ID
❌ Fields That CANNOT Be Indexed
  • Text Area (Long)
  • Text Area (Rich)
  • Non-deterministic formulas (see below)
  • Encrypted Text
  • Multi-select picklists (single-column index)
Non-Deterministic Formula Triggers
A formula field is non-deterministic if it:
  • References another entity via lookup (Acct.Owner.Name)
  • Uses TODAY(), NOW(), $User.*
  • References Owner / autonumber / division
  • References per-object special standard fields (Opp.Amount, Case.IsClosed, etc.)
  • References multi-select picklists or multicurrency currency
Workarounds — pick the right one for the case:
  • Cross-object reference? — Use dot-notation directly in WHERE (WHERE Acct__r.Other__r.Indexed__c = ...). As long as the leaf field is indexed, the cross-object path uses it. Best when the formula's only role is traversal.
  • Single-object derivation, hot filter? — Stamp the formula result into a real stored field via record-triggered Flow / before-save Apex on create-and-edit, then index the stored field. The recommended pattern at LDV scale; persistent fields outperform formulas every time.
  • Stamped field is also a unique business key? — Flag it External ID (or External ID + Unique). Salesforce auto-indexes External ID fields on Auto Number / Email / Number / Text — no Support case required, no XML deploy. Bonus: enables upsert() on that key.
  • Single-object deterministic formula you can't easily stamp? — Ask Support to custom-index the formula directly. Deterministic formulas (no TODAY(), no cross-entity refs, no special standards) can be indexed as-is. Salesforce automatically rebuilds the index if the formula changes.
  • Static lookup-style derivation (Region → Tier)? — Resolve in Apex from a Custom Metadata Type before the query; query by the resolved indexed field. No SOQL row cost, no formula at all.
  • Filter is rare / one-off? — Two-pass pattern: bounded indexed first query into an Id Set, then WHERE Id IN :ids AND ...formula stuff. See §12.
Full treatment with concrete code rewrites for each pattern: §10 — "Extremely Poor Pattern: Formula Fields in WHERE / Filter Clauses."
🗳 Getting a Custom Index Created — Two Paths, One Outcome
When To Request One (Decision Triggers)
  • A WHERE filter is provably selective (rollup query in §03 shows < 10% of rows match) but the field has no index

09 — Selectivity — What It Is, Why It Matters, and the Index Threshold Ladders
💡 Orientation — What Is Selectivity?
Definition (Salesforce-specific)
Selectivity is a numeric property of a filter — not of a field. It is the ratio:
selectivity = (rows matched by the filter) / (total rows on the object)
A filter is "selective" when it returns a small enough fraction of the object's rows that the optimizer (§03) is willing to use an index path to retrieve them. A filter is "unselective" when it returns too many rows — at which point the optimizer skips the index and falls back to a full driving-table scan, which on LDV objects means timeout territory.
Why The Ladders Exist
The line between "selective" and "unselective" is not a fixed percentage. It depends on:
  • Total row count on the object — bigger objects have tighter selectivity windows
  • Whether the index is standard or custom — standard indexes get more headroom (they're more efficient)
  • The absolute row cap imposed by the platform (1M for standard, 333K for custom) — even a "10% selective" filter on a 100M-row object isn't selective if 10% means 10M rows
The ladders below quantify this for every realistic object size. If your filter's match count fits in the right column, the index is used. If it doesn't, the index is skipped — even if the field is indexed.
Why this is the most important section in the document: every selectivity-driven anti-pattern (§10), every AND/OR/LIKE rule (§11), and every "should I ask Support for a custom index?" decision (§32) traces back to these threshold ladders. Architects who know these numbers diagnose LDV pain in seconds; those who don't write tickets that bounce around for weeks.

10 — Index-Defeating Query Patterns — Filters That Force a Full Scan
💡 Orientation — What "Index-Defeating" Means:
A field can be perfectly indexed and your filter can still cause a full table scan if the shape of the predicate denies the optimizer (§03) the ability to walk the index. This is not about which fields are indexed (§08) or how big the matched set is (§09) — this is about syntactic patterns that the index physically cannot serve. Each card below is a pattern that silently disables your index even when the field is indexed and the filter is selective. If a query is unexpectedly slow on an LDV object and SOQL Explain (§30) shows leadingOperationType: TableScan, this region is your first stop.
❌ Negative Operators — Always Full Scan
SOQLReports / List Views

NOT LIKE"does not contain"

NOT (positive expr)(any wrapped negation)
Why: Indexes are built on "what IS"; expressing "what is NOT" requires the optimizer to read every row to confirm the absence. There is no anti-index data structure.
Rewrites
// BAD
WHERE Status__c != 'Closed'
// GOOD — positive enumeration
WHERE Status__c IN ('New','Open','Pending')
❌ Leading Wildcards in LIKE

The 100K sample rule: For a non-leading-wildcard LIKE, the optimizer doesn't consult its stats table; instead it samples up to 100,000 actual rows to decide selectivity at run time.
For "contains" search at scale: use SOSL (§13). The search index is built for substring tokenized matches; SOQL is not.
❌ Range Comparisons on Text Fields
  • text_field < 'M'
  • text_field > 'A'
  • text_field <= 'X'
Lexicographic comparisons on text fields disable index usage even when the field is indexed. Only date and number ranges work.
Range collapse bonus: The optimizer detects date and number ranges on the same field and treats them as a single combined filter (CreatedDate >= X AND CreatedDate <= Y). Take advantage.
❌ NULL Comparisons on Indexed Fields
Single-column custom indexes exclude rows where the indexed column is NULL. So:
// BAD — full scan
WHERE Acct_Id__c = null
WHERE Status__c != null
Three Workarounds
  • Sentinel value: never let the field be NULL; default to 'N/A'.
  • Indexed formula field: IF(ISBLANK(Status__c), 'blank', Status__c) + index it.
  • Support case: ask for "custom index with nulls included"; existing indexes must be rebuilt to honor the change.
Two-column indexes CAN have NULLs in the second column even without the Support flag — useful for skinny composite filters.
⚠ EXTREMELY POOR PATTERN — Formula Fields in WHERE / Filter Clauses
The single most common LDV anti-pattern in poorly-audited orgs. A formula field is computed per-row at query time and is non-storable in the index table the way a real field is. Even when a formula looks like it should resolve to a simple value, the optimizer almost always falls back to a full scan when one appears in WHERE, in a Report filter, or in a List View filter. On a 10M-row object this turns a 200ms query into a 90-second timeout.

Mitigation Decision Tree
  • Single-object, deterministic, hot filter? → Pattern A: stamp the value into a real stored field (Flow/Apex on create/edit), then index the stored field.
  • Formula spans a lookup? → Pattern B: use cross-object dot notation directly in WHERE — the leaf field's index is used.

  • Filter is one-off / rare? → Pattern C: pre-resolve in Apex into an Id Set, then query WHERE Id IN :ids (the Two-Pass pattern, §12).
  • Filter pattern is IF(ISBLANK(x), 'sentinel', x)? → This formula IS deterministic; ask Support to index it.
Architectural rule: Formula fields are for display, calculations, and small-set automation. If a formula appears in a filter on an object > 100K rows, treat it as a refactor candidate. The fix is almost always Pattern A (stamp it) or Pattern B (dot notation).

& AND — Composite Index Path
Optimizer uses a composite index when an AND filter targets fewer than:
  • 2× the per-filter threshold for each filter (the "twice" rule), AND
  • The intersection threshold of the filters combined.

AND skip threshold> 20% OR 666,666 rows

📍 Filter Ordering — Should Indexed Fields Come First Left-to-Right?
The Honest Answer
For the Lightning Platform Query Optimizer, lexical left-to-right order in the WHERE clause does NOT determine the leading operation. The optimizer evaluates each filter against its statistics table, scores candidate paths by selectivity, and picks the most selective indexed predicate as the leading operation regardless of where it sits in the source SOQL. This is unlike older RDBMS where filter ordering was a tuning lever.
// These two are equivalent to the optimizer:
WHERE NotIndexed__c = 'X'
  AND CreatedDate >= LAST_N_DAYS:7   // indexed

WHERE CreatedDate >= LAST_N_DAYS:7     // indexed
  AND NotIndexed__c = 'X'
// Same plan: CreatedDate index leads in both.
But Lead With Indexed Filters Anyway — 5 Reasons
  • Code-review intent. A reader can see at a glance which filter is doing the selectivity work. WHERE CreatedDate >= ... AND Notes__c LIKE ... tells the reviewer "the date range carries the query."
  • Defensive against optimizer regression. Optimizer behavior has shifted across releases (Spring '14, '17, '20+ all introduced changes). Lexical ordering is a cheap insurance policy.
  • Apex Dynamic SOQL. When the WHERE clause is built dynamically by string concatenation, always emit the indexed selective filter first — it makes a buggy concatenation visible (no leading index = obvious problem).
  • External tools and exports. Workbench, Data Loader, integration adapters, Bulk API job queries, and some third-party tools may not always use the same optimizer reasoning. Lead with indexed fields so the query is portable.
  • It costs nothing. Reordering a WHERE clause has zero runtime cost. The defensive habit is free.

12 — Two-Pass Query Pattern — Salvaging Non-Selective Filters
✅ The Pattern in 3 Steps
  1. Pass 1 — Selective scope query. Use a known-indexed selective filter (date range, OwnerId, RecordType) to retrieve a bounded ID set. The optimizer always uses an index here.
  2. Build an in-memory ID set from Pass 1 (or pass it via Database.QueryLocator).
  3. Pass 2 — Targeted filter query. Apply the originally-non-selective predicate WHERE Id IN :idSet AND ...nonSelective.... Now the leading filter is the indexed Id set, and the non-selective branch only filters the bounded set.
Why this works: The leading filter (Id IN) is always index-driven. Whatever you do downstream is constrained to that bounded set, so even non-selective predicates run cheaply.
Concrete Apex Example
// Pass 1: bounded indexed scope
List<Id> recentIds = new List<Id>();
for (Customer_Product__c cp :
       [SELECT Id FROM Customer_Product__c
        WHERE CreatedDate = LAST_N_DAYS:7
        ORDER BY CreatedDate
        LIMIT 99999]) {       // < 100K activates ORDER BY index trick
  recentIds.add(cp.Id);
}

// Pass 2: apply the unselective predicate to bounded set
List<Customer_Product__c> matched =
  [SELECT Id, Product_Code__c
   FROM Customer_Product__c
   WHERE Id IN :recentIds
     AND NotIndexedFlag__c = true];
Salesforce's own tip: When using LAST_N_DAYS with ORDER BY on an indexed field and LIMIT < 100,000, the optimizer uses the ORDER BY index even if the filter is at the standard-index threshold.

14 — SOSL→SOQL Fallback & Cache-Latency Patterns — Not-Found Recovery
🔥 Architect Opinion — SOSL Is The Most Under-Used Performance Lever On The Platform.
An overwhelming majority of Salesforce Apex implementations reach for SOQL by reflex for record lookup, never even consider SOSL, and silently pay a 10×–100× performance tax on LDV objects as a result. Why this happens isn't entirely clear — the developer-onboarding ecosystem teaches SOQL as the default, the cross-object framing in older docs makes SOSL feel "specialized," and the 15-minute search-index lag is treated as a deal-breaker rather than a managed condition (§15 Platform Cache + the SOQL fallback below handle it cleanly). The platform's global tokenized search index is purpose-built for low-latency record retrieval and routinely outperforms relational-index SOQL by an order of magnitude on multi-million-row objects — especially for Name / Email / Phone / partial-text matches that would otherwise force unindexable LIKE '%term%' patterns. Leaving it on the table is the single largest unforced LDV performance error in the Salesforce-developer mainstream.
The architect's default rule: for any Apex record-lookup against an LDV object, evaluate SOSL FIRST. Only rule it out after a deliberate "not a good fit" check against the disqualifier list below.
✓ Use SOSL FIRST when (the typical case)
  • Looking up a record (or small set) by tokenizable text: name, email, phone, address, free-form description text, external reference codes
  • User-supplied search input (typeahead, lookup field, search bar, agent-driven case lookup)
  • Object size > ~100K rows, especially > 1M (where SOQL selectivity ladders start biting hard — §09)
  • The data is > 15 minutes old (i.e., already in the search index)
  • You'd otherwise reach for LIKE '%term%' — SOSL handles tokenized substring matching natively at search-index speed
  • "Find this thing if it exists" semantics — cache-and-fallback handles cold cache cleanly
  • Cross-object lookup where you don't know which object holds the data
✗ Rule SOSL Out (the disqualifier list — explicit checklist)
  • You need the record you JUST wrote in the same transaction or within ~15 minutes — the search index hasn't caught up. Use SOQL, or use the SOSL→SOQL fallback below to cover the lag window explicitly.
  • You need an aggregation, count, or sum — SOSL doesn't do COUNT(), GROUP BY, or SUM(). Use SOQL.
  • You need to filter on numeric / date / boolean ranges as the primary predicate (not as a RETURNING ... WHERE qualifier). Use SOQL.
  • You need strict ORDER BY with deterministic sort beyond what RETURNING ... WHERE ... ORDER BY provides. Use SOQL.
  • You need the result set as a related sub-query (parent ↔ child traversal, semi-join, anti-join). Use SOQL.
  • You need transactional read consistency across multiple records (e.g., reading sibling rows for a roll-up calculation that must reflect the same instant). Use SOQL.
  • The matched set on a single object will exceed 2,000 rows — SOSL silently truncates ("search crowding") past that. Use SOQL or use SOSL with a more specific RETURNING ... WHERE filter to stay under.
  • The field you're searching is not a search-indexed type (Name / Phone / Text / Picklist / long-text are; numeric / date are not). Use SOQL.
None of the above apply? Then SOSL is the right call. Pair with the cache-latency fallback pattern below to handle the only remaining gap (just-written records) and you have an LDV-scale lookup that performs at search-index speed.
The full read pattern for an LDV-scale UI lookup, in priority order:   (1) §15 Platform Cache hit (sub-ms) →   (2) SOSL with bounded RETURNING (search-index speed, ~15-min stale tolerance) →   (3) SOQL fallback on the lag window only (recent-records-only, indexed by CreatedDate) →   (4) Cache the result (positive AND negative). This is one of the strongest LDV read patterns available and is structurally invisible to most implementations.
1 · SOSL First (the fast path)
List<List<SObject>> r = [
  FIND :term IN NAME FIELDS
  RETURNING
    Account(Id, Name WHERE
      IsDeleted=false AND
      CreatedDate >= :since
      LIMIT 200),
    Contact(Id, Name WHERE
      IsDeleted=false
      LIMIT 200)
];
  • Tokenized + cross-object
  • Bounded by RETURNING
  • Misses records < 15 min old
2 · SOQL Fallback (catch the lag gap)
If SOSL returns zero results, check if the user just created a record SOSL hasn't indexed yet:
if (r[0].isEmpty() && r[1].isEmpty()) {
  // Lag-window SOQL: bounded by CreatedDate index
  DateTime lagWindow =
    DateTime.now().addMinutes(-20);
  results = [
    SELECT Id, Name FROM Account
    WHERE CreatedDate >= :lagWindow
      AND Name LIKE :term + '%'
    LIMIT 50
  ];
}
  • Bounded by CreatedDate (always indexed)
  • LIKE 'foo%' is index-eligible
  • Skip the leading-wildcard temptation
3 · Platform Cache for Hot Misses
Same term searched twice in 60 seconds — don't run SOSL twice. Cache the not-found marker too:
String key = 'srch:'+term;
List<Id> ids = (List<Id>)
  Cache.Org.get(key);
if (ids == null) {
  ids = doSoslThenSoql(term);
  Cache.Org.put(key,
    ids != null ? ids : new List<Id>(),
    300); // 5-min TTL incl. NEGATIVE
}
Cache the negative result. Every "not found" without caching is another full search. Negative caching with a short TTL is the real win.

15 — Platform Cache as LDV Pain Mitigation — Customer Cache Patterns
💡 Orientation — Why Platform Cache Belongs in an LDV Toolkit:
What it is
Platform Cache (a.k.a. Customer Cache) is an in-memory key-value store layered on top of the platform, accessible from Apex via Cache.Org and Cache.Session. Reads return in sub-millisecond time and bypass the database entirely — no SOQL row count, no governor cost beyond a tiny CPU charge, no sharing-engine evaluation, no statistics consultation.
Why it matters at LDV scale
Every cached lookup is one query the optimizer (§03) doesn't have to plan, one selectivity threshold (§09) doesn't have to clear, one sharing join doesn't have to fire, one row count doesn't accrue against your transaction's 50K SOQL budget. On hot read paths over LDV objects, cache hit rates of 80–95% can transform an unworkable design into a routine one.
The blunt rule: if you SOQL the same record more than once per minute org-wide, cache it. If you SOQL the same reference table thousands of times an hour, cache the lot. The first cache hit pays for itself.
🏠 Cache.Org — Organization-Wide Cache
  • Shared across all users in the org
  • Survives across user sessions and Apex transactions
  • Best for reference data, tier configs, ROLLUP results, externally-fetched data, expensive aggregates
  • Max TTL: 48 hours per entry
  • Default TTL if unspecified: 24 hours
  • Per-key serialized payload: 100 KB max
👤 Cache.Session — Per-User Session Cache
  • Scoped to a single user's session
  • Best for user-specific working state: pagination cursors, pre-computed permission decisions, expensive per-user queries
  • Auto-evicted when session ends
  • Max TTL: 8 hours per entry
  • Default TTL: 8 hours
  • Per-key serialized payload: 100 KB max
🔨 Allocation, Partitions & Edition Limits
Platform Cache is provisioned via partitions. Setup → Platform Cache → New Platform Cache Partition. Each partition has an explicit Org Cache + Session Cache MB allocation; the unallocated default partition is 0 MB — you must allocate before you can write.

A · Hot Reference Data
Lookup tables that everyone reads but nobody changes often: tier rules, region codes, fee schedules, picklist-driven configurations, mapping tables.

// Often a Map<Id, Account> you'd otherwise
// re-query in each trigger handler

  Cache.Org.get('myApp.hotAccts');

Combine with bulkified handler patterns (§05) to make trigger cascades cheap.
I · Negative Caching (Anti-Hammer)
Cache the absence of a result so repeated "not found" lookups don't keep hammering the DB.
// Use a sentinel for "we checked, it's not there"
final Object NEGATIVE = '__none__';
Object v = Cache.Org.get(key);
if (v == null) {
  Result r = expensiveQuery();
  Cache.Org.put(key,
    r != null ? (Object)r : NEGATIVE,
    300);
  return r;
}
return (v == NEGATIVE) ? null : (Result)v;
A repeated "not found" without negative caching is a hidden DOS vector against your own LDV objects. Always cache the empty result.
⚠ Pitfalls
  • Best-effort, not guaranteed. Platform Cache can evict any entry at any time under memory pressure. Always treat get() returning null as a normal cold path; never assume a recent put() is still live.
  • 100 KB serialized cap per key. Don't try to cache a full sObject list; cache summaries or chunked sub-keys.
  • Stale-write hazard. Two transactions can race on put(). If the cached value drives correctness (not just performance), use a versioned key or fall back to SOQL.
  • No transactional invalidation. If a record changes, your cache doesn't know. Use record-triggered Flow / Apex to remove() stale keys, or set short TTLs and accept staleness.
  • Default partition is 0 MB. Always create a named partition; never rely on the default.
✓ Best Practices
  • Namespace your keys. myApp.scope.identifier — protects against managed-package collisions.
  • Cache shape, not dump. Compute the projection (only the fields you need); cache that. Smaller payloads = more headroom.
  • Always provide a TTL. Don't rely on default TTL; set the shortest you can tolerate. Stale data is worse than slow data.
  • Monitor hit rate. Setup → Platform Cache → Diagnose. < 60% hit rate → payload too large or TTL too short.
  • Invalidate on write. If your data has known mutation paths, wire a Flow / trigger to Cache.Org.remove(key) on change. Cheap insurance.

📈 The Rule of 200 — Salesforce's Implicit Batch Constant
200 is the implicit batch size that pervades almost every record-processing surface in the Salesforce platform. It is not a single governor limit you can look up; it is a recurring architectural constant baked into how triggers fire, how Bulk API jobs split, how Batch Apex defaults, how the SOAP API caps each call, and how the implicit Apex SOQL for-loop streams. Internalize it and the iteration patterns below stop feeling like a collection of tricks and start feeling like the natural shape of the platform.

The pattern is so consistent that "200" is shorthand among Salesforce architects for "the unit of bulk." When someone says "your trigger isn't bulkified," they mean it doesn't tolerate being called with all 200 records present in Trigger.new.
Why 200 (and not 100, or 1,000)
  • Memory pressure — heap usage scales with batch size; 200 sObjects fits comfortably under the 6 MB sync / 12 MB async heap cap (§06) for typical record shapes
  • Lock duration — lock contention scales linearly with batch size; 200-record DML windows have proven empirically safe across most LDV workloads (§18)
  • Per-call overhead amortization — small enough to fit transactional limits, large enough that the platform's per-call cost (parsing, sharing eval, validation) is amortized across meaningful work
  • Governor budget headroom — leaves room within the per-transaction 100 SOQL / 50K-row / 150 DML / 10K-DML-row caps for handler logic to do follow-up work without blowing limits
⚠ Bulk API 2.0 — "batch size" means the upload chunk, NOT the processing chunk:
Job orchestration / upload unit
When you submit a Bulk API 2.0 ingest job, the API accepts up to 10,000 records per CSV upload "batch". This is a job-orchestration knob — it controls how the API queues, splits, and monitors your job's records. It does not control how those records are subsequently processed against your org.
Trigger / record-processing unit
When the platform actually processes the uploaded records, it internally chunks back down to 200-record windows for each trigger fire, sharing evaluation, and DML execution. A 10,000-record upload becomes 50 chained 200-record trigger invocations. Heap, locks, and governor accounting apply per 200-record window — not per 10,000-record upload.
Practical consequence: there is no Bulk API knob that makes triggers see a larger Trigger.new than 200. Tuning the upload batch size only changes job-orchestration latency and parallelism, not the per-handler invocation shape. Bulkified handler design is not optional — it's the only design. Salesforce documentation describes the 10,000 figure as the upload-batch maximum without claiming it as a per-handler scope; the once-floated "raise batch size for performance" advice was always about job orchestration, not processing chunks.
Architectural Implications — What "Bulkified" Actually Means
✓ Bulkified handler shape
  • Handler tolerates being called with any size of Trigger.new, from 1 to 200
  • Handler tolerates being called multiple times in one transaction (chained 200-record fires from a 1,000-record DML)
  • All SOQL filtered by Trigger.newMap.keySet() — never inside a per-record loop
  • All DML aggregated into one call per object type per handler invocation — never per-record
  • Static recursion guard (a Set<Id> of already-processed Ids) prevents the same record from being touched in a chain
✗ Common 200-boundary anti-patterns
  • "Tested with 50 records, looks fine" — the 200-record case has different behavior; lock and governor profiles differ qualitatively
  • SOQL or DML inside a for loop over Trigger.new — 200 records × 1 SOQL each = 200 SOQL queries (limit is 100)
  • Static List/Set not reset between chained 200-record fires — leaks state, blows heap on the second fire
  • DML on Trigger.new without sorting/grouping by parent — lock contention storms (§21)
  • Assuming a Bulk API 2.0 10K-record upload batch becomes a 10K-record Trigger.new — it doesn't; the platform still serializes through 200-record trigger windows. Code expecting larger windows breaks at the first prod load
⚙ What "1,000-Record DML" Actually Looks Like Inside the Platform
A single update accts; on 1,000 records doesn't fire your trigger once with 1,000 rows. It fires it five times sequentially with 200 rows each — same transaction, same static state, fresh budget per fire. Code that "tested fine on 50 records" hits five new failure modes here.
Mental model: when designing any record-bearing operation on a Salesforce LDV object, ask "how does this behave when it receives exactly 200 records?" and "how does this behave when it is called five times in a row with 200 records each?" — if you can answer both confidently, your code will scale. If you can't, it won't.

17 — USING SCOPE & WITH USER_MODE / SYSTEM_MODE
💡 Orientation — Two Different SOQL Levers, Often Confused:
USING SCOPE <filter>
Narrows the universe of rows the optimizer considers, by leveraging an indexed visibility predicate (OwnerId, role hierarchy, queue membership, territory, etc.). It's the programmatic equivalent of the list-view scope picker. The query still runs through the sharing engine; USING SCOPE just gives the optimizer a smaller starting set.
WITH USER_MODE / WITH SYSTEM_MODE (Spring '23+)
Per-query toggle for whether sharing, FLS, and CRUD are enforced on the read. USER_MODE = full enforcement; SYSTEM_MODE = bypass entirely. Replaces the older class-level with sharing / without sharing declaration with a finer-grained inline control.
Why both matter for LDV: on a 20M-row Case object with a deep role hierarchy, the sharing engine becomes the dominant cost of every read. USING SCOPE Mine + SYSTEM_MODE together can drop a multi-second query to milliseconds — the SCOPE narrows the universe via OwnerId index, and SYSTEM_MODE skips the sharing engine entirely. They are orthogonal levers that compose; understanding the difference is the difference between solving the symptom and naming the right control.

18 — Record Locking — The Additive Principle
🔒 Locks Compose — You Hold More Than You Think
Salesforce takes locks to ensure referential integrity. A single DML can hold multiple locks — row + parent + sharing + lookup + roll-up master + group table.
Worked Example: One Opportunity Insert
  1. Row lock on the new Opportunity
  2. Lock on parent Account (Opportunity inserts always lock parent)
  3. If a roll-up summary on Account — extra lock on Account (regardless of whether the value changes)
  4. If a custom lookup with "Don't Allow Deletion" — lock on the looked-up record
  5. If private sharing model — share-row locks
  6. If account has community/portal users in role — Group Table lock on owner change
🔒 Locks Acquired by ONE Opportunity Insert (Worst-Case Configuration)
A single typed insert opportunity; can hold 5–7 locks simultaneously when the org has the worst-case configuration (RollUp summary on Account + non-clearing lookup + private OWD + portal users in role hierarchy). Each red-bordered box is a lock the platform takes for you, invisibly. The Group Table lock (rightmost) is the one most likely to serialize the entire org.
Reasoning rule: When estimating lock contention, walk every row in the chain the DML touches. Don't reason about the typed object alone.
When Locking Becomes A Real Problem
  • LDV objects (millions of rows)
  • Large bulk loads / API integrations
  • High concurrency on a hot parent (account skew)
  • Group membership / role / territory ops in parallel
  • Owner reassignment of accounts with portal users
  • FOR UPDATE queries with wide row sets
The wait timeout: Salesforce's lock wait is approximately 10 seconds; on timeout you get UNABLE_TO_LOCK_ROW. (Not exposed by Salesforce as a configurable, but treat it as the upper bound.)

21 — UNABLE_TO_LOCK_ROW — Recovery & Prevention Patterns
Root Causes (in order of frequency)
  • Bulk DML not grouped by parent — multiple chunks fight over same Account/Campaign row
  • Concurrent integrations targeting hot parents
  • Account ownership change while Cases/Opps are being inserted
  • Roll-up summary on a hot parent — every child write locks it
  • Trigger that does follow-up DML on parent records
  • "Don't Allow Deletion" lookup pointing at a single hot record (skew)
Prevention Patterns
  • Group bulk DML by ParentId — Salesforce's #1 documented mitigation
  • Sort child-insert batches so each batch hits one parent at a time
  • Reduce SOAP batch size (default 200 → 100 or 50) for hot parents
  • Use serial mode for Bulk API jobs touching hot parents
  • Switch lookups from "Don't Allow Deletion" to "Clear the value"
  • Run portal user provisioning single-threaded
  • Schedule heavy automation off-business-hours
  • Distribute child records across dummy parent buckets (§22)

⚠ Use only as last resort. Same-transaction retries:
  • Burn the SAME governor budget (DML rows, CPU, heap)
  • Block the user thread; can hit the >5s long-running limit
  • Retry the WHOLE batch even though only a subset locked
  • No backoff between attempts (retries collide with the same lock)
For LDV-grade resilience, prefer the Async Queueable Retry pattern below.
✅ Recommended — Async Queueable Retry (production-grade, scales)
Why It Beats Synchronous
  • Allow-partial-success DML (Database.insert(rs, false)) lets the non-locked rows commit; only the locked subset is collected for retry
  • Per-row error categorization — "unable to obtain exclusive access" goes to retry; everything else escalates as a real error
  • Re-enqueues only the locked subset — each retry pass works on a smaller set, naturally converging
  • Fresh governor budget per Queueable invocation — CPU, heap, DML rows reset
  • Doesn't block user thread — original transaction commits and returns immediately
  • Queueable-depth guard — falls back to synchronous if depth is exhausted, never silently drops work
  • JSON clone isolation — JSON.deserialize(JSON.serialize(rs), List<SObject>.class) prevents trigger-context reference-list contamination
Pattern attribution: Adapted from the production-grade DMLQueueable + QueueableUtility pair (Coleman, 2017–2018), refined across multi-million-row Salesforce deployments. See §16 for the complementary streaming-iteration patterns.
The Pattern (compressed)
public class DMLQueueable implements Queueable {
  public enum Operation { INSERTS, UPDATES, DELETES }
  private SObject[] records;
  private Operation op;
  private String ctx;

  public DMLQueueable(SObject[] rs, Operation o, String c) {
    // JSON clone — isolate from trigger reference list
    this.records = (SObject[])JSON.deserialize(
        JSON.serialize(rs), List<SObject>.class);
    this.op = o; this.ctx = c;
  }

  public void execute(QueueableContext ctx) {
    SObject[] lockRetries = new SObject[0];
    String[] realErrors = new String[0];

    // Allow partial success — key second arg = false
    Database.SaveResult[] results =
      (op == Operation.INSERTS)
        ? Database.insert(records, false)
        : Database.update(records, false);

    for (Integer i = 0; i < results.size(); i++) {
      if (results[i].isSuccess()) continue;
      for (Database.Error err : results[i].getErrors()) {
        if (err.getMessage()
              .contains('unable to obtain exclusive access')) {
          // Lock-row error — retry async
          lockRetries.add(records[i]);
        } else {
          // Real error — escalate, don't retry
          realErrors.add(err.getMessage());
        }
      }
    }

    if (!realErrors.isEmpty()) Logger.error(realErrors, ctx);
    if (!lockRetries.isEmpty()) {
      // Re-enqueue ONLY the lock-failed subset
      QueueableUtility.handleQueueable(
        new DMLQueueable(lockRetries, op, ctx));
    }
  }
}

// Depth-aware launcher with sync fallback
public class QueueableUtility {
  public static void handleQueueable(Queueable job) {
    try {
      if (Limits.getQueueableJobs()
              < Limits.getLimitQueueableJobs()) {
        try { System.enqueueJob(job); }
        catch (AsyncException ae)  { job.execute(null); }
        catch (LimitException le) { job.execute(null); }
      } else {
        job.execute(null);   // chain depth exhausted
      }
    } catch (Exception e) { Logger.exception(e); }
  }
}
When To Use
  • Trigger / Flow handler doing follow-up DML on a hot parent
  • Integration adapters writing to LDV objects under concurrent load
  • Mass owner reassignment / sharing recalc cascade
  • Anywhere "unable to obtain exclusive access" appears intermittently
Caveats
  • Only retries lock-row errors — validation/required-field/permission errors must be handled separately
  • Asynchronous — the calling user doesn't know retry is happening; use Platform Events to notify if needed
  • Test with Test.startTest()/stopTest() to flush the queue exactly once
  • Debug logs split across each Queueable invocation
What This Doesn't Replace
  • Root-cause fixes (skew, ordering, grouping by parent) — §22
  • Bulk-API serial-mode for hot-parent loads — §23
  • Lookup-deletion-behavior changes that eliminate the lock entirely — §20
Resilience ≠ design. The async retry handles transient lock contention gracefully, but the org should not depend on it for normal operation. Treat persistent retry traffic as a signal to investigate skew or grouping.
22 — Data Skew — The Three Forms & Their Mitigations
💡 Orientation — What "Data Skew" Means & Why It Hurts:
Skew is the opposite of distribution. When records pile up unevenly — one parent holding tens of thousands of children, one user owning every imported lead, one lookup target referenced by 80% of rows — that pile becomes a serialization point. Every DML on a child record locks the hot parent (§18); every owner change cascades through the sharing engine; every concurrent insert collides on the same lookup target. The org gets slower the more it succeeds at the very thing the user is doing well. The threshold is consistent across all three forms: 10,000. Past that, you are in skew territory.
📊 Healthy Distribution vs. Skewed Pile-Up — Same Total, Vastly Different Behavior
Both populations contain the same 10,000 records. The healthy population spreads the load, scales linearly, and never serializes. The skewed population locks the hot parent on every concurrent write — UNABLE_TO_LOCK_ROW, sharing-recalc storms, owner-change cascades. Salesforce's "10,000" threshold is the line above which the platform's locking and sharing math collapses.

25 — Delete Strategies — Soft Delete, Hard Delete, Truncate
Soft Delete & Recycle Bin
  • Default delete() is a soft delete (flag + Recycle Bin)
  • 15 days retention before scheduled hard delete
  • Doesn't count against storage
  • No limit on Recycle Bin row count
  • Soft-deleted rows still affect query performance (still in heap)
  • Reports/queries must explicitly filter IsDeleted=false on multi-join
Standing rule: Keep the Recycle Bin empty when running multi-join reports on LDV objects. Salesforce explicitly cites it as a query-performance factor.
Hard Delete (Bulk API 2.0)
  • Available via Bulk API 2.0 + Bulk API "hard delete" job operation
  • Bypasses Recycle Bin entirely
  • Records become immediately available for physical purge
  • Required to manage profile permission: "Bulk API Hard Delete"
Recommended For
  • Deleting > 1,000,000 records
  • Periodic archive purges
  • Cleaning up bad imports
Order of deletion: Always delete children first, then parents. Master-detail cascades work but trigger more locks.
Truncate Custom Object (Sandbox Only)
For sandbox cleanup: Truncate instantly empties a custom object — faster than mass delete, no Recycle Bin involvement.
  • Custom objects only
  • Sandbox only (Setup → Object Manager → Truncate)
  • Cannot truncate if: master-detail child of standard object, has Custom Indexes, Big Object source, External Lookup target, in MD relationship that is the parent, or has External ID/Unique fields
Production mass-delete? Use Bulk API 2.0 Hard Delete with PK Chunking. Contact Support only if Bulk API performance is insufficient or if structural cleanup needs metadata-level intervention.

A · Skinny Tables
A platform-managed, denormalized copy of frequently-accessed fields. No soft-deleted rows. No join between standard-fields-table and custom-fields-table at the DB level.
Limits
  • Max 100 / 200 columns per skinny table (verify per Salesforce reference)
  • Cannot include fields from other objects
  • Read-only by definition
  • Auto-syncs with source on changes
Allowed Field Types
Checkbox, Currency, Date, Date/time, Email, Number, Percent, Phone, Picklist (incl. multi-select), Text, Text Area (incl. long), URL, encrypted

Account, Contact, Opportunity, Lead, Case + custom objects

B · External IDs
Marking a field as External ID auto-creates an index. Marking External ID + Unique enables upsert() by external key.
Allowed Field Types
  • Auto Number
  • Email
  • Number
  • Text
  • NOT picklist, formula, currency, date, etc.
The 6-Per-Object Cap
Hard limit: 6 External IDs per object. Architectural reasons: each is a maintained index. Hit the cap?
Going Beyond 6 (Support Path)
  • Open a case with Salesforce Support
  • Justify the need (per-system integration keys)
  • Provide query plan / use cases
  • Often granted up to 15 per object
Better alternative: a single composite key field "<system>:<id>" on a single External ID field, with the system enum as part of the key. Reduces fragmentation.
C · Two-Column & Null-Inclusive Indexes
Two-Column Custom Index
  • When two fields recur in the same WHERE
  • Beats two single-column indexes for AND filters
  • Allows NULL in second column by default
  • Useful for filter+sort patterns: WHERE State='CA' ORDER BY City
Null-Inclusive Custom Index
  • Single-column custom indexes exclude NULLs by default
  • Ask Support to create / convert to "include nulls"
  • Existing index must be rebuilt — cannot just toggle a flag
  • Useful when business logic legitimately uses NULL as a value
Architects' decision rule: Two-column index for AND queries that always co-occur. Null-inclusive only when redesigning the data model is impossible.

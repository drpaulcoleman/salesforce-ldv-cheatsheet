01 — Why Migrate from Heroku
📉 The Heroku® Platform Situation (2022–2025)
Several years after its Salesforce® acquisition, Heroku has now entered a more or less maintenance-only posture, no longer selling enterprise-class platform subscriptions. Some key events that signal migration urgency, none of which reflects official Salesforce documentation but rather the author's own observations:
YearEventImpact
2022Free tier eliminated for all dynos, Postgres, RedisHigh — Forced all hobby apps to pay
2023Heroku Data partners (third-party Postgres/Redis)Medium — SLA/pricing uncertainty
2024Eco dyno replaces free; no new platform featuresMedium — No container-native roadmap
2025Pricing increases across Shield/Enterprise tiers; CI in maintenanceHigh — Cost at scale doubles
Bottom line: Heroku is stable for existing workloads but has no strategic investment path for enterprise customers. Only the Team-level subscription plans can be setup online in self-service ways. And while existing Heroku Enterprise-class customers can continue to use the platform at their current subscription terms, teams building for enterprise scale should contemplate an exit window that realigns to cloud computing platforms that benefit from sustained capital investment and innovation.
🎯 When to Stay vs. When to Move
✓ STAY on Heroku if…
App traffic is predictable and low
Team has no cloud/K8s ops expertise
Add-ons (Postgres, Redis) meet SLA needs
Monthly bill < $300 and not growing
Startup runway is short (< 6 months)
✗ MIGRATE if…
Heroku Postgres plan limits hit (storage/connections)
Need multi-region deployment or custom VPC
Bill exceeds $500/month (GCP is 40–60% cheaper)
Container-native CI/CD is a team requirement
Compliance requires private networking for DB/cache
Migration Trigger Checklist: If you tick 2+ of the "migrate" boxes, begin your evaluation. A typical migration window for a mid-sized Node.js app takes 2–4 weeks with the right tooling (and significantly less with AI assistance — see Section 17).

🗄 Postgres Migration Key Fact: The hardest part is not the data export — it's the dual-write window. Run old and new DB in parallel for 15–30 minutes during cutover, with your app writing to both. Zero data loss is achievable with this pattern.
🔴 Redis Migration Key Fact: Do not export Redis data. Redis is a cache. Your app will re-warm it naturally within minutes of startup. The only migration task is updating the REDIS_URL to the new private IP.
🪣 Storage Migration Key Fact: Copy objects with gsutil rsync (from S3 to GCS) before DNS cutover. Update your SDK from aws-sdk/S3 to @google-cloud/storage. The bucket naming convention can stay the same.
🔒 Heroku ACM vs GCP Google-Managed SSL — How TLS Handling Differs
Heroku ACM: Provisions a Let's Encrypt cert automatically on heroku domains:add. TLS terminates at Heroku's edge routers — your dyno only sees HTTP internally. CNAME your domain to a Heroku DNS target (e.g. tokyo-1234.herokudns.com). Cert is per-app; auto-renewed at ~60 days. Custom upload: heroku certs:add cert.pem key.pem.
GCP Google-Managed SSL: Provisions a Google Trust Services cert automatically on the Global HTTPS LB. TLS terminates at GCP's Load Balancer — Cloud Run receives HTTP/2 internally. Point your DNS A record to the LB IP (gcp static load balancer ip assigned to your apps).
The cert is per-LB-frontend, attached to the target HTTPS proxy — not the Cloud Run service directly. Custom upload: gcloud compute ssl-certificates create --certificate=cert.pem --private-key=key.pem.
✅ Similarity: Both platforms fully automate cert provisioning and renewal — zero manual cert management. DNS propagation (5–30 min) is the only wait time on both platforms.  |  ⚠️ Key difference: Heroku cert = one cert per app. GCP cert = one cert per LB frontend, shareable across all backend services (dev/staging/prod can use separate forwarding rules with their own certs, or a multi-domain cert).

05 — Phase 0: Prerequisites & GCP Project Setup
Step 0.1 — Enable Required APIs
Run once per GCP project. Takes ~2 minutes.
# Authenticate and set project
gcloud auth login
gcloud config set project YOUR_PROJECT_ID

# Enable all required APIs in one command
gcloud services enable \
  run.googleapis.com \
  sqladmin.googleapis.com \
  redis.googleapis.com \
  secretmanager.googleapis.com \
  storage.googleapis.com \
  cloudbuild.googleapis.com \
  artifactregistry.googleapis.com \
  cloudscheduler.googleapis.com \
  servicenetworking.googleapis.com \
  logging.googleapis.com \
  monitoring.googleapis.com
vs. Heroku: heroku create myapp — one command. GCP setup is more verbose but each API is independently billable and auditable.
Step 0.2 — Create Service Accounts
# Cloud Run runtime identity
gcloud iam service-accounts create app-run \
  --display-name="App Cloud Run SA"

# Grant roles to Cloud Run SA
gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \
  --member="serviceAccount:app-run@YOUR_PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/cloudsql.client"

gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \
  --member="serviceAccount:app-run@YOUR_PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"

# Cloud Scheduler identity (invoker only)
gcloud iam service-accounts create app-scheduler \
  --display-name="App Scheduler SA"
Heroku has no equivalent. Heroku manages service identity internally. GCP separates identity from config — a security advantage.
Step 0.3 — Container Registry Setup
# Create Artifact Registry repo for Docker images
gcloud artifacts repositories create app-images \
  --repository-format=docker \
  --location=us-central1 \
  --description="App container images"

# Configure Docker auth for the registry
gcloud auth configure-docker us-central1-docker.pkg.dev

# Heroku equivalent: git push heroku main (buildpack auto-builds)
# GCP: build is decoupled — you control the image
Key difference: Heroku buildpacks auto-detect your language and build a slug. GCP requires a Dockerfile (see Phase 7). This gives you full control but requires a one-time conversion effort.
GCP best practice: build once, promote the same image digest through dev → staging → prod. Never rebuild for promotion — same image hash guarantees identical behavior.
06 — Phase 1: Heroku Postgres → Cloud SQL
🗄 Migration Overview Moderate Complexity
What changes
  • ✓ DATABASE_URL value (new Cloud SQL private IP)
  • ✓ Connection must go through VPC (private IP only)
  • ✓ ssl: {rejectUnauthorized: false} may need update
What stays the same
  • ✓ Full Postgres 15 compatibility
  • ✓ All schemas, tables, indexes, RLS policies
  • ✓ Same pg / postgres npm drivers
Step 1.1 — Export from Heroku Postgres
# Capture a fresh backup on Heroku
heroku pg:backups:capture --app myapp-prod

# Download the backup as a .dump file
heroku pg:backups:download -o db-export.dump --app myapp-prod

# Or: export via pg_dump directly (requires DATABASE_URL)
pg_dump -Fc -v $DATABASE_URL -f db-export.dump

# Verify the dump is not empty
ls -lh db-export.dump   # Should be > 0 bytes
Step 1.2 — Create Cloud SQL Instance
# Create Cloud SQL PostgreSQL 15 instance (production)
gcloud sql instances create myapp-db-prod \
  --database-version=POSTGRES_15 \
  --tier=db-g1-small \
  --region=us-central1 \
  --no-assign-ip \
  --network=myapp-vpc \
  --availability-type=REGIONAL

# Create the app database
gcloud sql databases create myapp --instance=myapp-db-prod

# Set the postgres user password
gcloud sql users set-password postgres \
  --instance=myapp-db-prod --password=STRONG_PASSWORD

# Get the private IP address (store for DATABASE_URL)
gcloud sql instances describe myapp-db-prod \
  --format="get(ipAddresses[0].ipAddress)"
Step 1.3 — Import Data & Update Connection String
# Upload dump to GCS (Cloud SQL needs GCS source)
gsutil cp db-export.dump gs://myapp-backups/db-export.dump

# Import via Cloud SQL (handles pg_restore internally)
gcloud sql import sql myapp-db-prod \
  gs://myapp-backups/db-export.dump \
  --database=myapp

# OR: use pg_restore directly via Cloud SQL Auth Proxy
# Start proxy locally: cloud-sql-proxy PROJECT:REGION:INSTANCE
pg_restore -h 127.0.0.1 -U postgres -d myapp db-export.dump

# New DATABASE_URL format for Cloud SQL (private IP)
DATABASE_URL=postgresql://postgres:PASSWORD@10.100.x.x/myapp?sslmode=disable

# Store in Secret Manager
echo "postgresql://postgres:PASSWORD@10.100.x.x/myapp" | \
  gcloud secrets create myapp-prod-DATABASE_URL --data-file=-
Private IP only: Memorystore and Cloud SQL are only accessible from within the VPC. Your Cloud Run service must have Direct VPC Egress enabled. See Phase 8.
07 — Phase 2: Heroku Redis → Memorystore
🔴 Migration Overview Simple — No Data Export
Redis is a cache, not a database. You do NOT export or migrate Redis data. Your application will re-warm the cache naturally within minutes of connecting to the new instance. The only task is provisioning Memorystore and updating REDIS_URL.
What changes
  • ✓ REDIS_URL → private IP (no redis://user:pass@host)
  • ✓ Must use VPC — no public endpoint
  • ✓ AUTH disabled by default (VPC-only security)
What stays the same
  • ✓ Full Redis 7 compatibility
  • ✓ Same ioredis / redis npm clients
  • ✓ All Redis commands, pub/sub, streams
Step 2.1 — Create Memorystore Redis Instance
# Create Memorystore for Redis (Standard HA tier)
gcloud redis instances create myapp-cache-prod \
  --size=1 \
  --region=us-central1 \
  --redis-version=redis_7_0 \
  --tier=STANDARD_HA \
  --network=myapp-vpc

# Wait for provisioning (~5 minutes), then get private IP
gcloud redis instances describe myapp-cache-prod \
  --region=us-central1 \
  --format="get(host)"
# Returns: 10.0.1.34 (example private IP)

# Heroku equivalent:
# heroku addons:create heroku-redis:mini -a myapp-prod
Step 2.2 — Update REDIS_URL Secret
# Store the new Redis private IP in Secret Manager
echo "redis://10.0.1.34:6379" | \
  gcloud secrets create myapp-prod-REDIS_URL --data-file=-

# Verify: read back the secret value
gcloud secrets versions access latest \
  --secret=myapp-prod-REDIS_URL

# Heroku equivalent:
# heroku config:get REDIS_URL -a myapp-prod
No AUTH password in the URL. Memorystore on a private VPC doesn't need Redis AUTH by default — the VPC firewall is your security boundary. If your app connects with redis://user:pass@host, remove the credentials.
Step 2.3 — Verify Connectivity from Cloud Run
// Node.js: test Redis connection (ioredis)
import Redis from 'ioredis';

const redis = new Redis(process.env.REDIS_URL);

redis.ping().then(r => {
  console.log('Redis ping:', r); // PONG
  redis.quit();
});
VPC required: This test only works when the code runs inside a Cloud Run service with Direct VPC Egress enabled. You cannot connect to Memorystore from your local machine without a VPN or Cloud SQL Auth Proxy. See Phase 8 for VPC setup.
08 — Phase 3: Papertrail → Cloud Logging
📋 Migration Overview Simple
No log drain setup needed. Cloud Run automatically captures all stdout / stderr output to Cloud Logging. JSON-structured logs are auto-parsed — no agent or sidecar required.
Papertrail
  • ✓ Log drain (HTTPS or syslog)
  • ✓ 1-day search on free plan
  • ✓ Simple text log search
  • ✗ No structured query support
  • ✗ No alerting policies
  • ✗ Extra add-on billing
Cloud Logging
  • ✓ Automatic — zero config on Cloud Run
  • ✓ 30-day retention (free tier: 1GB/day)
  • ✓ Structured log queries (Logging Query Language)
  • ✓ Log-based alerting policies
  • ✓ Export to BigQuery or GCS for analytics
  • ✓ Integrated with Cloud Trace and Monitoring
Step 3.1 — Switch to Structured JSON Logging
Cloud Logging automatically parses JSON with these reserved field names:
// Node.js: Pino logger setup (recommended)

Step 5.1 — Bulk Export & Import
# 1. Export all Heroku config vars to a .env file
heroku config -s --app myapp-prod > prod.env
# prod.env contains: KEY=VALUE lines (one per var)

# 2. Bulk import to Secret Manager (bash loop)
while IFS='=' read -r key value; do
  [[ "$key" == "#"* ]] && continue  # skip comments
  echo -n "$value" | gcloud secrets create myapp-prod-${key} \
    --data-file=- --replication-policy=automatic 2>/dev/null || \

13 — Phase 8: VPC & Private Networking (The Complex Part)
Why VPC? Cloud SQL and Memorystore have no public IP — by design. Your Cloud Run service must route traffic through a VPC to reach them. This is the most complex phase but also the biggest security upgrade over Heroku. AI assistance (Section 17) makes this dramatically easier.
Step 8.1 — Create VPC & Subnets
# Create the VPC network
gcloud compute networks create myapp-vpc \
  --subnet-mode=custom

# Subnet for Cloud Run egress traffic
gcloud compute networks subnets create subnet-run \
  --network=myapp-vpc \
  --region=us-central1 \
  --range=10.0.0.0/24

# Subnet for Cloud SQL (via Private Service Access)
gcloud compute networks subnets create subnet-db \
  --network=myapp-vpc \
  --region=us-central1 \
  --range=10.0.1.0/24
Step 8.2 — Private Service Access (for Cloud SQL)
# Allocate IP range for Google managed services
gcloud compute addresses create psa-range \
  --global \
  --purpose=VPC_PEERING \
  --prefix-length=16 \
  --network=myapp-vpc

# Create VPC peering connection to Google APIs
gcloud services vpc-peerings connect \
  --service=servicenetworking.googleapis.com \
  --ranges=psa-range \
  --network=myapp-vpc

# Cloud SQL will now get a private IP in the peered range
# Heroku: no equivalent — Postgres was always on public internet with SSL
Step 8.3 — Direct VPC Egress (Cloud Run → VPC)
# Option A: Direct VPC Egress (recommended, no extra cost)
# Add --vpc-egress flag when deploying Cloud Run
gcloud run deploy myapp-api-prod \
  --vpc-egress=all-traffic \
  --subnet=subnet-run \
  --network=myapp-vpc \
  --region=us-central1 ...

# Option B: Serverless VPC Connector (legacy, still supported)
gcloud compute networks vpc-access connectors create myapp-connector \
  --region=us-central1 \
  --subnet=subnet-run \
  --min-instances=2 --max-instances=3

gcloud run deploy myapp-api-prod \
  --vpc-connector=myapp-connector ...
Direct VPC Egress is the modern approach (2024+). It routes traffic natively through Cloud Run's network interface — no VPC connector billing, lower latency.
Step 8.4 — Cloud NAT (Outbound Internet from VPC)
# Cloud Run with VPC egress loses direct internet access
# Cloud NAT restores outbound internet (e.g. for SendGrid HTTPS)

# Create a Cloud Router
gcloud compute routers create myapp-router \
  --network=myapp-vpc --region=us-central1

# Create Cloud NAT on that router
gcloud compute routers nats create myapp-nat \
  --router=myapp-router --region=us-central1 \
  --auto-allocate-nat-external-ips \
  --nat-all-subnet-ip-ranges

# Optional: static NAT IP (for third-party IP allowlisting)
gcloud compute addresses create myapp-nat-ip --region=us-central1
gcloud compute routers nats create myapp-nat \
  --nat-external-ip-pool=myapp-nat-ip ...
Static NAT IP is useful when third-party APIs (payment processors, KYC services) require IP allowlisting. One IP covers all Cloud Run instances — no per-dyno IP management.
14 — Phase 9: Heroku Scheduler → Cloud Scheduler + Cloud Run Workers
⏰ Architecture: Workers Pattern
Heroku Scheduler runs node workers/emailWorker.js directly on a one-off dyno. Cloud Scheduler sends an authenticated HTTP POST to a Cloud Run workers service. Each worker becomes a route handler instead of a standalone process. 
// workers/index.js — HTTP router for all workers
// Handles POST /workers/{jobName} from Cloud Scheduler
import express from 'express';
import { runEmailWorker }  from './emailWorker.js';
import { runSyncWorker }   from './syncWorker.js';

const app = express();
app.use(express.json());

app.post('/workers/emailWorker', async (req, res) => {
  try {
    await runEmailWorker();
    res.json({ ok: true });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

app.post('/workers/syncWorker', async (req, res) => {
  await runSyncWorker();
  res.json({ ok: true });
});

app.listen(process.env.PORT || 8080);
Deploy as separate Cloud Run service with --no-allow-unauthenticated. Only Cloud Scheduler (via OIDC token) can invoke it.
Creating Cloud Scheduler Jobs
# Get the workers service URL
WORKERS_URL=$(gcloud run services describe myapp-workers-prod \
  --region=us-central1 --format="get(status.url)")

# Create a scheduler job (runs daily at 8am UTC)
gcloud scheduler jobs create http myapp-prod-emailWorker \
  --location=us-central1 \
  --schedule="0 8 * * *" \
  --uri="${WORKERS_URL}/workers/emailWorker" \
  --http-method=POST \
  --oidc-service-account-email=app-scheduler@PROJECT.iam.gserviceaccount.com \
  --oidc-token-audience="${WORKERS_URL}"

# Manually trigger a job to test
gcloud scheduler jobs run myapp-prod-emailWorker --location=us-central1

# Heroku equivalent:
# heroku run node workers/emailWorker.js -a myapp-prod  (one-off)
# heroku scheduler -- adds via dashboard UI only
OIDC is mandatory for Cloud Scheduler → Cloud Run workers. Without it, any HTTP client could trigger your workers. The --no-allow-unauthenticated flag on the workers service + OIDC token creates a cryptographically-verified auth chain.

15 — Phase 10: DNS Cutover — Zero-Downtime Domain Transfer
Pre-Cutover Checklist & Strategy
  1. 1
    Lower TTL 48h before: Change DNS TTL to 60 seconds. This ensures fast propagation when you make the actual switch.
  2. 2
    Provision GCP HTTPS LB: Create the Global HTTPS Load Balancer with your custom domain and Google-managed SSL cert before cutover.
  3. 3
    Test via /etc/hosts: Add GCP_LB_IP yourdomain.com to your local hosts file and verify the app works end-to-end on GCP.
  4. 4
    Dual-write database: During cutover window (15–30 min), configure the app to write to both Heroku Postgres and Cloud SQL. Ensures zero data loss.
  5. 5
    Switch DNS A record: Change domain's A record from Heroku IP to GCP LB IP. With 60s TTL, propagation is near-instant.
  6. 6
    Monitor for 30 minutes: Watch Cloud Logging for errors. Keep Heroku app running for 24h rollback window.
  7. 7
    Remove Heroku custom domain: Only after confirming GCP is healthy and stable.
DNS Cutover Commands
# ── HEROKU SIDE: Remove custom domain ──────────────
# Get Heroku's IP for your domain (for reference)
heroku domains --app myapp-prod

# ── GCP SIDE: Set up HTTPS Load Balancer ───────────
# Reserve a global static IP
gcloud compute addresses create myapp-lb-ip --global
gcloud compute addresses describe myapp-lb-ip --global --format="get(address)"
# → e.g. 34.111.x.x  (set this as your DNS A record)

# Create Google-managed SSL cert (auto-renews)
gcloud compute ssl-certificates create myapp-ssl-cert \
  --domains="api.yourdomain.com,www.yourdomain.com" \
  --global

# ── DNS: Change A record at your DNS provider ───────
# Before: api.yourdomain.com  A  54.x.x.x  (Heroku router IP)
# After:  api.yourdomain.com  A  34.111.x.x  (GCP LB IP)
# TTL: 60 (set 48h before cutover)

# ── VERIFY: cert provisioning (takes 10–60 min) ─────
gcloud compute ssl-certificates describe myapp-ssl-cert \
  --global --format="get(managed.status)"
# ACTIVE = cert issued, TLS working

# ── ROLLBACK: point DNS back to Heroku if needed ────
# Restore old DNS A record (old Heroku IP)
# heroku domains:add api.yourdomain.com -a myapp-prod
FIDO2 / WebAuthn continuity: If your app uses FIDO2 passkeys or WebAuthn, the Relying Party ID (rpId) is bound to the domain. As long as your custom domain doesn't change, all enrolled credentials continue to work without re-registration.

💡 Best Practices for AI-Assisted Migration
  1. 1
    Give full context upfront: Tell the AI your exact stack (Node version, ESM/CJS, framework, which Heroku add-ons, GCP region). Better context → better output on first attempt.
  2. 2
    Paste actual output, not descriptions: Share the output of heroku config -s, heroku ps, cat Procfile directly. AI works on real data, not summaries.
  3. 3
    Ask for explanations: "Explain every gcloud flag you used." Understanding what was generated helps you debug if something doesn't work as expected.
  4. 4
    Security review pass: After AI generates IAM roles or secrets config, ask: "Review this for security issues or over-permissive grants." AI can audit its own output.
  5. 5
    Iterative validation: Run one phase at a time. Paste error output back into the chat. AI can diagnose gcloud errors and suggest fixes without you leaving the terminal.
  6. 6
    Generate tests: Ask AI to generate connectivity tests (can Cloud Run reach Memorystore? Can it access Secret Manager?) to validate each phase before moving on.
Never share secrets with AI: Always redact API keys, passwords, and credentials before pasting config output into any AI chat. Use placeholder values like REDACTED or *** in shared snippets.
